📋このシステムは何をしているのか

JRAの競馬開催日（土日）に、レースのオッズを時系列で自動記録するシステムです。

「発走24分前のオッズはいくつだったか」「3分前はどう変わったか」を 9つの決まったタイミングで取得してDBに積み上げていきます。すべてcronが自動で動かします。

オッズ取得は 2ルート並行 して行います。

・JRAルート → importOdds が担当。…_odds テーブルに保存。
・ネットケイバルート → importRaceResult が担当。…_netkeiba_odds テーブルに保存。

両ルートが共同で書き込む共有テーブルとして t_horse_odds_finder_odds_get_timing が新設されました。「いつ・どちらが取得したか」を後勝ちで記録します。

odds_get_timing の役割

upsertキーは (date, kaisuu, basho, day, race, timing) の6カラム。
odds_from と get_datetime は後から書いた方が勝つ（後勝ち）設計。
JRAとnetkeibaの両ルートが同じ行を書き合い、最後に取得した方が残ります。

⏰cronの設定

サーバー上の実際の crontab -l 設定です。

# 土曜5:50 全テーブルをTRUNCATEする
50 5 * * 6     keiba:deleteKeibaTableRecords

# 毎日6:00 スケジュール・レース・馬情報を取得（土日のみ実質動作）
0 6 * * *      keiba:importSchedule

# 土日6:20 ネットケイバからレース一覧を取得する
20 6 * * 0,6   keiba:importRace

# 土曜7:00 土日両日分のオッズを初回取得する
0 7 * * 6      keiba:importBaseOdds

# 毎分9〜17時 JRAオッズを定点取得する
* 9-17 * * *   keiba:importOdds

# 毎分9〜17時 ネットケイバオッズを定点取得する
* 9-17 * * *   keiba:importRaceResult

⚙️6つのコマンドの役割

deleteKeibaTableRecords お掃除係
土曜 5:50 に1回だけ

前の週のデータを全部消します。7テーブルをTRUNCATEし、ログファイル6本を削除します。これが土曜の「スタート合図」です。

TRUNCATE × 7テーブル（odds_get_timing も含む）
ログファイル × 6本 削除
importSchedule 時刻表係
毎日 6:00（木・金は即終了）

Node.jsでスケジュールをスクレイピングし、schedules・races・horsesの3テーブルに保存します。トランザクション保護済み。

土曜は土日両日分をINSERT。日曜は日曜分だけDELETE→INSERT（土曜データは触らない）。

土曜6:00の失敗は致命的です。 racesテーブルが空になり、以降の全コマンドが「対象レース0件」で終了します。ログを必ず確認してください。
importRace ネットケイバ用の時刻表係
土日 6:20

ネットケイバのオッズ取得に必要な「race_id」付きレース情報を取得して専用テーブルに保存します。updateOrInsertのため再実行しても重複しません。

importBaseOdds 7時の一斉撮影係
土曜 7:00 に1回だけ

土日全レースのオッズを一括取得して minutes_before_start=999 で保存します。odds_get_timing への書き込みはありません（9時以降の定点取得から記録が始まります）。

importOdds JRA用の定点観測係
毎分 9:00〜17:59

毎分起動し、[24, 21, 18, 15, 12, 9, 6, 3, 0] のタイミングに合致したレースだけ処理します。1レース分の書き込みはトランザクションで保護されており、全馬のオッズ保存 + odds_get_timingへの記録がセットで成功/ロールバックします。

importRaceResult ネットケイバ用の定点観測係
毎分 9:00〜17:59

importOddsと同じ仕組みでnetkeibaのオッズを取得します。こちらもトランザクション保護済み。odds_get_timingへの後勝ちupsertもセットで実行されます。

🔢タイムスタンプの意味

値意味書き込み元
9997時のベースライン、または発走24分前時点のオッズ（upsertで更新）importBaseOdds / importOdds
21〜3各タイミングのスナップショット（追記）importOdds / importRaceResult
-999発走0分前の最終確定オッズimportOdds / importRaceResult

odds_get_timing の timing も同じ値体系。ただしこのテーブルは1レース×1タイミング=1行で、馬ごとの行は持ちません。

🚨失敗パターン別シミュレーション

パターン①：Node.js スクレイピング失敗

$output = shell_exec($command);
$odds   = json_decode($output, true);
if (!$odds) {
    $this->error("取得失敗。スキップします。");
    continue;  ← DB::transaction に到達しない
}
→ DBへの書き込みは一切なし。
oddsテーブルへの書き込みも、odds_get_timingへの書き込みも発生しません。
次のタイミング（3分後）に自動でリトライされます。

パターン②：DB書き込み中にエラー（トランザクション内）

importOdds / importRaceResult（共通）
BEGIN TRANSACTION
 馬1〜12: INSERT 成功
 馬13: DBエラー → 例外発生
 馬14, 15: 実行されない
 odds_get_timing: 実行されない
ROLLBACK → 馬1〜12のINSERTも全て取り消し
→ このタイミングのデータは0行。中途半端なデータは残りません。
次の3分後のタイミング（別の minutes_before_start）で取得が続きます。

パターン③：importRaceResult の -999 二重取得ガード

// handle() 内（トランザクション外）← 1重目のガード
if ($diffMinutes === -999) {
    $alreadySaved = exists(netkeiba_odds + -999)?
    if ($alreadySaved) continue;  // 既に保存済みならスキップ
}

// saveOdds() 内（トランザクション内）← 2重目のガード
if ($minutesBefore === -999) {
    $alreadySaved = exists(key + -999)?
    if (!$alreadySaved) insert();  // 存在しない場合のみINSERT
    return;
}
cronが重なって2プロセス同時に走っても、2重ガードにより -999 の重複INSERTは実質的に発生しません。

パターン④：土曜6:00のimportSchedule失敗

Node.js が空出力 or JSONパース失敗
  → return 1; で中断（DB::transaction に入らない）
  → 全テーブルの既存データはそのまま（土曜は空のまま）
致命的です。
racesテーブルが空のまま → importBaseOdds（7時）: 「対象レース0件」で終了 → importOdds/importRaceResult（9時〜）: 「対象レース0件」で毎分終了。

土曜6:05頃に tail importSchedule.log で「正常終了」を確認してください。

2026-05-30 実際に発生したサーバーダウンの原因分析。
「RAM 960MB が少なすぎた」のではなく、プロセスが際限なく積み重なる構造になっていたことが本当の問題でした。

修正前：連鎖崩壊のメカニズム

9:XX分 0秒 — importOdds 起動（1回目）
Chromium 起動 → +600MB

JRAサイトにアクセスしてオッズを取得しようとする。

shell_exec('node keibaOddsGetTanpuku.mjs ...')
↑ タイムアウト指定なし。JRAが遅ければ無限待ち。

PHP が shell_exec() でハング

Node.js は page.goto() の応答を待ち続ける。
Chromium は起動したまま 600MB を消費し続ける。
importRaceResult も同様に別の Chromium をハング中。

9:XX+1分 — cron が importOdds を再起動（2回目）
PHP ロックなし → 2つ目の PHP + 別のレースに新 Chromium

前のPHPがハング中でも、cronは1分後に容赦なく次を起動する。

2回目のPHPは前と別のレースを処理しようとするため、Node.jsロックをすり抜けて新しいChromiumを起動する。

RAM計算：
ベースライン 340MB + Chromium×2（1,210MB）+ active PHP×2（100MB）= 1,650MB
RAM 960MB を大幅超過。Swapなし → OOM キラー発動
9:XX+2分 — さらに3回目のimportOdds + importRaceResult
Chromium が 4〜6 本に増殖 → 完全にメモリ枯渇

OOMキラーが MySQL・Nginx を kill → DB接続・HTTP応答が消滅。

さくらVPSのコントロールパネルから強制リセットしか手段がない状態。

修正後：5つの防衛ラインの動き

① PHP レベルのロックファイル
/tmp/keiba_importOdds.lock が存在する間、次のcron起動は即終了する。
→ 同時に動くimportOddsは常に1本だけ。 
if (file_exists($lockFile)) {
    // 即終了。Chromium 起動なし。
    return;
}
② timeout 120（PHP側）
JRAサイトが完全に応答しなくなっても、120秒後にNode.jsプロセスを強制終了する。
→ PHPが永久ブロックすることがなくなった。 
shell_exec('timeout 120 node keibaOddsGetTanpuku.mjs ...')
③ try/finally でbrowser.close()を保証（Node.js側）
例外・timeout・正常終了、どのルートを通っても必ずChromiumが閉じる。
→ プロセスが残留してメモリを食い続けることがない。 
} finally {
    if (browser) await browser.close();
    if (existsSync(lockFile)) unlinkSync(lockFile);
}
④ Node.js レベルのロックファイル（スタールロック対応済み）
同じレースの Chromium が2本起動しない。
OOMキルでロックが残っても、PIDの生存確認で古いロックを自動削除して再起動できる。 
// OOMキル後の復旧

}
