④ importSchedule（毎日6:00）

ファイル: ImportKeibaSchedule.php + keibaOddsGetSchedule.mjs

1
多重起動防止チェック（keiba_importSchedule.lock）
2
曜日チェック: 木曜（4）・金曜（5）はスキップして終了
3
keibaOddsGetSchedule.mjs を Node.js で実行（timeout 300秒）
stderr → keibaOddsGetSchedule.log にリダイレクト、stdout のみ PHP が受け取る

keibaOddsGetSchedule.mjs の動作

1
Playwright / Chromium 起動（headless）
2
JRA トップページ（jra.go.jp）にアクセス → 「オッズ」リンクをクリック → 開催選択ページへ
3
h3.sub_header から開催情報（日付・回数・場所・何日目）を一覧取得
4
各開催をループ：開催リンクをクリック → レース一覧ページへ
5
各レースをループ：単複リンクをクリック → 馬名・騎手・調教師・馬番・枠番を取得 → レース一覧へ戻る
6
schedules / races / horses の3配列を JSON で stdout に出力
7
ブラウザ・ロックファイルをクローズ（finally ブロックで確実に実行）

PHP 側の後処理（JSON受け取り後）

1
JSON パース失敗ならエラーログを出して return 1
2
出走済みチェック: start_time === 'XXX' のレースが当日分全てなら終了
3
土曜：データを絞り込まず土日両日分をそのまま使う
土曜以外：当日分のみに絞り込む
4
トランザクション開始
5
挿入対象日付の schedules / races / horses / odds を DELETE（べき等性確保）
6
schedules を INSERT → races を INSERT（start_time===XXX はスキップ）→ horses を INSERT
7
コミット（途中失敗は自動ロールバック）
日曜6:00の注意点：
上記ステップ5で t_horse_odds_finder_odds の日曜分も DELETE される。
土曜7:00に importBaseOdds が保存した日曜分の基準オッズ（999）もここで消える。
日曜分の基準オッズは importOdds が diff===24 のタイミングに再取得・保存する。

⑦ importOdds（毎日9:00〜17:59 毎分）

ファイル: ImportKeibaOdds.php + keibaOddsGetTanpuku.mjs + keibaOddsGetWide.mjs

1
多重起動防止チェック（keiba_importOdds.lock）— 前回が終わっていない間は次のcronが来ても即終了
2
t_horse_odds_finder_races から当日レース一覧を発走時刻順に取得
3
各レースをループ開始
4
発走時刻と現在時刻の差（$diff）を計算（四捨五入）
$diff < 0（発走済み）はスキップ
$diff が ODDS_GET_TIMING に含まれない分数もスキップ
5
【単複】keibaOddsGetTanpuku.mjs 実行（timeout 120秒、最大2回リトライ）
6
【単複】t_horse_odds_finder_odds に保存（トランザクション）
diff===24 → minutes_before_start=999（UPSERT）
diff===0 → minutes_before_start=-999（INSERT）
それ以外 → minutes_before_start=diff値（INSERT）
7
【単複】t_horse_odds_finder_odds_get_timing に取得日時・odds_from='JRA' を UPSERT
8
getWideOdds($race) を呼び出し
9
【ワイド】keibaOddsGetWide.mjs 実行（timeout 120秒、最大2回リトライ）
10
【ワイド】t_horse_odds_finder_odds_wide に保存（トランザクション）
minutes_before_start のマッピングは単複と同じ
2026-06-03修正 INSERT → UPSERT（重複INSERT防止）
11
次のレースへ → ループ繰り返し

処理時間の注意：
単複 最大120秒 + ワイド 最大120秒（各リトライ込み）= 1レース最大約5分。
同じタイミングで合致するレースが複数あると後続レースの取得が遅延することがある。
多重起動防止ロックで二重実行は防いでいるため、データが壊れることはない。

⑩ 注意点・確認事項

🟢 Wide INSERT → UPSERT 修正済み（2026-06-03）

修正済み t_horse_odds_finder_odds_wide への保存を常時 INSERT から UPSERT（exists→update / else→insert）に変更。
同一キー（date / kaisuu / basho / day / race / uma1 / uma2 / minutes_before_start）のレコードが重複することがなくなった。

🟢 Wide ログファイルパス修正済み

修正済み getWideOdds() のログパスを base_path('/var/www/.../importOddsWide.log') ではなく
'/var/www/horse_odds_finder/storage/logs/importOddsWide.log' の絶対パスで直接指定済み。
base_path() に絶対パスを渡すと二重パスになる問題を回避。

🟢 既存処理（単複）への影響なし

単複の取得・DB保存ロジックはコードを変更していない。ワイドは単複保存の後続処理であるため、ワイドが失敗しても単複の結果には影響しない。

🟢 keibaOddsGetWide.mjs 動作確認済み

2026-06-03 にサーバで手動実行済み。2回東京12日 1R で 120組 のオッズ取得・JSON出力を確認。

🟡 1レースあたりの処理時間

単複 最大120秒 + ワイド 最大120秒 × 各2回リトライ = 最大約5分/レース。
取得タイミング合致レースが複数重なると後続レースの取得が遅延する可能性がある。
多重起動防止ロック（keiba_importOdds.lock）があるため二重実行は防げる。

🟡 スタールロック対策（keibaOddsGetTanpuku / Wide）

ロックファイルが残存していても isProcessAlive(pid) でPIDが死んでいれば古いロックを自動削除する。
ただし keibaOddsGetSchedule.mjs にはこの対策が未実装（PIDチェックなし）。
importSchedule が途中でkillされた場合、次回起動時に keibaOddsGetSchedule.lock が残って空データが返る可能性がある。サーバ再起動時には手動でロックファイルを削除すること。
rm /var/www/horse_odds_finder/scripts/keibaOddsGetSchedule.lock

⑪ MJS スクリプトの戻り値（JSON形式）

keibaOddsGetSchedule.mjs → stdout

schedules / races / horses の3配列を1つのオブジェクトで返す。PHP が JSON パースして各テーブルに INSERT する。
{ "schedules": [ { "date": "2026-06-07", "kaisuu": "2", "basho": "05", "basho_name": "東京", "day": 12 } ], "races": [ { "date": "2026-06-07", "kaisuu": "2", "basho": "05", "basho_name": "東京", "day": 12, "race": 1, "race_name": "3歳未勝利", "start_time": "10:05:00", "num_horses": 16 } ], "horses": [ { "date": "2026-06-07", "kaisuu": "2", "basho": "05", "basho_name": "東京", "day": 12, "race": 1, "waku": 1, "num": 1, "name": "ホース名", "horse_url": "https://www.jra.go.jp/...", "jockey": "騎手名", "trainer": "調教師名" } // ... 全レース全馬分 ] }

keibaOddsGetWide.mjs → stdout

ワイドの組ごとのオッズ配列。16頭立てで最大120組。uma1 が caption（軸の馬番）、uma2 が各行の相手馬番。全フィールドが文字列。
