# 競馬オッズ取得システム 動作解説

最終確認日: 2026-05-28

---

## このシステムは何をしているのか

JRAの競馬開催日（土日）に、レースのオッズを時系列で自動記録するシステムです。

「発走24分前のオッズはいくつだったか」「3分前はどう変わったか」を
9つの決まったタイミングで取得してデータベースに積み上げていきます。
人間が手動で操作する必要はなく、すべてサーバー上のスケジューラー（cron）が自動で動かします。

オッズの取得は2ルート並行して行います。
- **通常ルート**（JRAサイト）: `importOdds` が担当
- **ネットケイバルート**: `importRaceResult` が担当

2つのルートで取ったデータはそれぞれ別のテーブルに保存されます。

---

## 登場する6つのコマンドの役割

### 1. `deleteKeibaTableRecords`（お掃除係）
**いつ動くか**: 土曜 5:50 に1回だけ

前の週のデータが全部残ったままだと困るので、週の始まりに全テーブルを空にします。
「TRUNCATE（トランケート）」という操作で、6つのテーブルを一括で空っぽにします。
同時に、前の週のログファイル（6種類）も削除して、ログが古いデータで汚れないようにします。

このコマンドが終わった時点でデータベースは完全な白紙状態です。
これが土曜の一連の処理の「スタート合図」になります。

---

### 2. `importSchedule`（時刻表係）
**いつ動くか**: 毎日 6:00（木・金は即終了）

Node.js スクリプトを呼び出して、競馬の開催スケジュールをスクレイピングします。
取得した内容を3つのテーブルに保存します。

- **schedules テーブル**: どこの競馬場で何回開催か
- **races テーブル**: 各レースの発走時刻・頭数・レース名
- **horses テーブル**: 各レースの出走馬一覧（馬名・騎手・調教師など）

**土曜と日曜で動きが違います。**

土曜は「deleteKeibaTableRecords がすでに全部消してくれている」ので、
そのまま土日両日分のデータをまとめて INSERT します。

日曜は土曜のデータを消さないよう気をつけながら、日曜分だけを入れ替えます。
「日曜分を DELETE してから INSERT」という順番で、
途中でエラーが起きてもデータが壊れないようにトランザクション（まとめて成功・失敗する仕組み）で囲んでいます。

木・金曜日は競馬開催がないため、コードの先頭で曜日を判定して即終了します。
月・火・水曜日はコードが動きますが「今日のレースは0件」と判断して何もせず終わります。

---

### 3. `importRace`（ネットケイバ用の時刻表係）
**いつ動くか**: 土日 6:20

`importSchedule` と目的は似ていますが、ネットケイバ専用のデータを取得します。
ネットケイバのオッズを取得するには「race_id」という固有のIDが必要で、
それを `t_horse_odds_finder_netkeiba_races` テーブルに保存するのがこのコマンドの仕事です。

6:00 の `importSchedule` が終わってから動く（6:20）のは、
`importSchedule` が登録した「今週の開催日付」を読んで、
その日付に対応するレースだけを取得する設計だからです。

---

### 4. `importBaseOdds`（7時の一斉撮影係）
**いつ動くか**: 土曜 7:00 に1回だけ

土日全レースのオッズをまとめて取得します。
「週末が始まる前の、最初の記念撮影」のようなものです。

このコマンドが取得したオッズは `minutes_before_start = 999` という値で保存されます。
999 は「発走よりかなり前の早い時点のオッズ」を意味する特別な値です。

土曜のレースも日曜のレースも、この時点でまとめて取得します。
（日曜分については後述する理由でいったん削除されますが、設計上問題ありません）

---

### 5. `importOdds`（JRA用の定点観測係）
**いつ動くか**: 毎分 9:00〜17:59

毎分起動して、「今この瞬間、処理すべきレースがあるか」を確認します。

ポイントは**全レースを毎回処理するわけではない**ことです。
あらかじめ決めた9つのタイミングに合致したレースだけを処理します。

```
$ary = [24, 21, 18, 15, 12, 9, 6, 3, 0]
```

各レースについて「発走まで何分か（小数点切り捨て）」を計算して、
この配列に含まれる分数のときだけ Node.js を呼び出してオッズを取得します。

たとえば毎分起動しても「今は残り23分のレースしかない」という場合はスキップして終わります。
これによって、全体の処理量を最小限に抑えています。

---

### 6. `importRaceResult`（ネットケイバ用の定点観測係）
**いつ動くか**: 毎分 9:00〜17:59

`importOdds` とほぼ同じ仕組みで、ネットケイバのオッズを取得します。
取得先と保存先のテーブルが違うだけで、タイミングのロジックは全く同じです。

---

## オッズが保存されるタイムスタンプの意味

各オッズデータには「何分前のデータか」を示す `minutes_before_start` という値が記録されます。

| 値 | 意味 |
|---|---|
| `999` | 「かなり前の段階のオッズ」を示す特別な値。7時のベースライン、または発走24分前時点のオッズ。 |
| `21` | 発走21分前のオッズ |
| `18` | 発走18分前のオッズ |
| `15` | 発走15分前のオッズ |
| `12` | 発走12分前のオッズ |
| `9` | 発走9分前のオッズ |
| `6` | 発走6分前のオッズ |
| `3` | 発走3分前のオッズ |
| `-999` | 「発走直前の最終オッズ」を示す特別な値。発走0分前（ぎりぎり直前）のオッズ。 |

レース1本・馬1頭あたり、最大9行のオッズデータが記録される計算になります。

## 1週間の動き（時系列シミュレーション）

### 月・火・水曜日

cron は動きますが、実質的には何も起きません。

6:00 の `importSchedule` は Node.js を呼んでスケジュールを取得しますが、
「今日（月〜水）のレース」に絞り込むと0件になるため、データの追加も削除もしません。

9:00〜17:59 の `importOdds` と `importRaceResult` も、当日のレースが0件なので即終了します。

---

### 木・金曜日

6:00 の `importSchedule` が曜日を判定して即終了します。
その他のコマンドも当日レース0件で即終了します。
サーバー的には何も起きていません。

---

### 土曜日（最も重要な日）

#### 5:50 — まず全部リセットする

前の週のデータを完全に消します。

```
6テーブル全て TRUNCATE（空っぽに）
6つのログファイルを削除
```

これ以降の処理は全て「空のデータベース」に書き込んでいきます。

---

#### 6:00 — 今週の情報を登録する

Node.js がJRAサイトから今週の開催情報を取得します。
土曜なので、土日両日分のデータをまとめて保存します。

```
t_horse_odds_finder_schedules  ← 土日の開催スケジュール
t_horse_odds_finder_races      ← 土日の全レース一覧（発走時刻・頭数など）
t_horse_odds_finder_horses     ← 土日の全出走馬情報
```

この時点でシステムは「今週、どのレースが何時に走るか」を把握した状態になります。

---

#### 6:20 — ネットケイバ用の情報を登録する

`t_horse_odds_finder_schedules` に登録された日付をもとに、
ネットケイバから race_id 付きのレース情報を取得します。

```
t_horse_odds_finder_netkeiba_races  ← 土日の全レース（race_id付き）
```

`importRaceResult` はこのテーブルを参照して動くため、
このコマンドが先に終わっている必要があります。
6:20 というスタート時刻はそのための余裕です。

---

#### 7:00 — 全レースのオッズを一斉取得する

土日全レースのオッズを Node.js で一括取得して `minutes_before_start = 999` で保存します。

```
t_horse_odds_finder_odds  ← 土日全レース・全馬の999オッズ
```

レース数が多い（土日合わせて60レース前後）ため、処理に数分かかる場合があります。
終了時にログにサマリーが出ます。

```
処理レース数 : 60 件
保存頭数合計 : 900 頭
失敗レース数 : 0 件
処理時間     : 87.3 秒
```

---

#### 9:00〜17:59 — 毎分、タイミングが合ったレースだけ処理する

毎分クロンが起動して、全レースについて「今は発走何分前か」を計算します。

**計算方法**: `発走時刻 - 現在時刻` を秒で求め、60で割って切り捨て → 整数の残り分数

この残り分数が `[24, 21, 18, 15, 12, 9, 6, 3, 0]` のどれかに一致したとき、
そのレースのオッズを取得します。一致しない分数はスキップします。

**10:00発走のレースを例に取ると:**

| 時刻 | 残り分数 | $ary に含まれるか | 処理 |
|---|---|---|---|
| 9:36 | 24 | ✅ | 7時に入れた999行を最新オッズで**更新** |
| 9:37 | 23 | ✗ | スキップ |
| 9:38 | 22 | ✗ | スキップ |
| 9:39 | 21 | ✅ | 新しく1行**追加**（21分前オッズ） |
| 9:40〜41 | 20, 19 | ✗ | スキップ |
| 9:42 | 18 | ✅ | 新しく1行追加 |
| 9:45 | 15 | ✅ | 新しく1行追加 |
| 9:48 | 12 | ✅ | 新しく1行追加 |
| 9:51 | 9 | ✅ | 新しく1行追加 |
| 9:54 | 6 | ✅ | 新しく1行追加 |
| 9:57 | 3 | ✅ | 新しく1行追加 |
| 9:58〜59 | 2, 1 | ✗ | スキップ |
| 10:00 | 0 | ✅ | 新しく1行追加（-999 = 最終オッズ） |
| 10:01〜 | -1以下 | — | 発走後なので永久にスキップ |

**なぜ24分前だけ「更新」で、他は「追加」なのか**

7時のベースオッズと24分前オッズはどちらも `999` という同じ値で保存されます。
仮に単純に「追加」してしまうと、同じレース・同じ馬に999の行が2行できてしまいます。
そこで24分前だけは「すでに999行があれば更新、なければ追加」という処理をしています。

---

### 日曜日

#### 6:00 — 日曜分だけ入れ替える

土曜のデータはそのままに、日曜分だけを更新します。

```
t_horse_odds_finder_schedules (日曜分) → 削除→再登録
t_horse_odds_finder_races     (日曜分) → 削除→再登録
t_horse_odds_finder_horses    (日曜分) → 削除→再登録
t_horse_odds_finder_odds      (日曜分) → 削除（土曜7時に取得していた日曜の999も消える）
```

土曜7時に取得した「日曜分の999ベースオッズ」がここで消えます。
これは仕様上の動作で、日曜9時以降に `importOdds` が各レースの発走24分前に
新しい999を取得するため、実運用上は問題ありません。

**土曜のデータは一切触りません。**
土曜のオッズや馬情報はそのまま残ります。

#### 6:20 — ネットケイバ用の情報を更新する

土日両日付について netkeiba_races テーブルを更新します。
土曜レースの情報も更新されますが、`importRaceResult` は「今日の日付」で絞って動くので影響はありません。

#### 9:00〜17:59 — 土曜と全く同じ流れ

土曜と同じ仕組みで、日曜分のレースを処理します。
発走24分前の最初の処理では999行がまだない（6時に削除済み）ので、新規追加から始まります。

## トラブルが起きたときどうなるか

### Node.js のオッズ取得に失敗した場合

`importOdds` と `importRaceResult` は1レースの取得に失敗してもそのレースをスキップして次へ進みます。
次のタイミング（3分後）に再び試みます。

`importBaseOdds` は失敗したレースを集計して終了時にログへ出力します。

```
失敗レース数 : 2 件 ← 要確認
  [FAIL] 東京 5R (2026-05-30)
  [FAIL] 阪神 3R (2026-05-30)
```

### importSchedule が失敗した場合

Node.js の出力が取れなかった場合、エラーをログに出して処理を中断します。
この場合、既存のデータは削除されず前の状態のままです（トランザクションでロールバックされます）。

### 同じ処理が重なって実行された場合

`importOdds` の処理が1分以上かかって次のクロンと重なっても問題ありません。
「同じレース・同じタイミング」を2回処理することはなく、
残り分数が変わるたびに別の行が作られる設計のため重複しません。
（24分前の999だけは upsert なので絶対に重複しません）

---

## 2つのオッズルートの関係

このシステムには「JRAルート」と「ネットケイバルート」の2系統が並走しています。

| | JRAルート | ネットケイバルート |
|---|---|---|
| レース情報の取得 | `importSchedule` | `importRace` |
| オッズの取得 | `importOdds` | `importRaceResult` |
| レーステーブル | `t_horse_odds_finder_races` | `t_horse_odds_finder_netkeiba_races` |
| オッズテーブル | `t_horse_odds_finder_odds` | `t_horse_odds_finder_netkeiba_odds` |

2系統は完全に独立しています。片方が失敗してももう片方には影響しません。

## 修正した問題点の記録

### 致命的なバグ（修正済み）

**`importKeibaRace.php` と `importKeibaRaceResult.php` — Node.js のパス指定が不完全だった**

cron はサーバーの「通常の検索パス」を使わず、フルパスで実行ファイルを指定しないと見つけられません。
`node` とだけ書いてあったため、手動実行では動くのに cron から動かすと必ず失敗する状態でした。

```
修正前: node /path/to/script.mjs
修正後: /home/centos/.nvm/versions/node/v24.15.0/bin/node /path/to/script.mjs
```

### 重要なバグ（修正済み）

**`ImportKeibaOdds.php` — 7時のベースオッズと24分前オッズが重複していた**

7時の `importBaseOdds` が999を INSERT したあと、
9時以降の `importOdds` が発走24分前に同じレース・同じ馬の999をもう1行 INSERT していました。
その結果、同じデータが2行できる状態になっていました。

修正後は「999がすでに存在すれば更新、なければ追加」という処理に変わっています。

### その他の修正（修正済み）

| ファイル | 内容 |
|---|---|
| `ImportKeibaSchedule.php` | PHP 的に不正な終了処理（`exit()`）を正しい書き方（`return 0`）に修正 |
| `ImportKeibaSchedule.php` | Node.js が完全に無応答だったときにエラー判定が甘かった部分を修正 |
| `ImportKeibaBaseOdds.php` | 処理時間の計測に精度の低い方法を使っていたため、正確な方法に変更 |
| `importKeibaRace.php` | 計算結果を変数に入れたまま一度も使っていないコードを削除 |

## 最終判定

✅ **全コマンド問題なし。土日の本番稼働に耐えられる状態です。**

把握しておくべき仕様:

- **日曜レースの7時ベースオッズは日曜の6時に消える**: 設計上の動作で、9時以降に再取得されるため問題なし。
- **発走後のオッズは取得しない**: -999（発走0分前）が最後の記録になる。発走後のデータは収集対象外。
